# AGENTS.md

## Project Mode

This project is a Mixed project.

- The user and agents may both edit code.
- Agents must assume user changes can appear at any time.
- Do not overwrite, revert, or clean up user work unless explicitly asked.
- If user changes conflict with agent work, stop and explain the conflict before continuing.

## Development Method

Use agile iteration for this project.

- Do not apply hyper-waterfall by default.
- Prefer short inspect -> change -> verify loops.
- Keep plans lightweight and update them as the code and product shape become clearer.
- When requirements are unclear, ask focused questions instead of expanding scope.
- Deliver small working increments over large speculative designs.

Hyper-waterfall may be used only when the user explicitly asks for it.

## Permission Model

Agents may do the following without additional approval:

- Read files.
- Search the codebase.
- Analyze structure and behavior.
- Suggest features, fixes, or improvements.
- Ask clarification questions.

Agents must request approval before:

- Creating, modifying, moving, or deleting files.
- Refactoring existing code.
- Creating new directories.
- Running shell commands that execute processes, tests, builds, crawlers, installers, or scripts.
- Installing dependencies.
- Performing any action that changes project state.

When approval is required, state the intended action, why it is needed, and what files or commands are involved.

## Working Style

- Think before coding.
- State assumptions when they affect the implementation.
- Surface tradeoffs when there is more than one reasonable interpretation.
- Prefer the simplest implementation that satisfies the current requirement.
- Avoid abstractions for single-use behavior.
- Do not add configurability unless it is requested or clearly needed.
- Keep changes surgical and tied to the current request.
- Match the existing project style.

Every changed line should trace back to the user request or to cleanup caused by the agent's own change.

## Product Contract From SPEC.md

NTIndex is a static site generator for NeonTeam model-swap videos.

Expected title pattern:

```text
A as B | Game Model Swap
```

Example:

```text
Furina as Nahida | Genshin Impact Model Swap
```

Core data flow:

```text
YouTube -> Crawler -> SQLite -> Builder -> JSON -> HTML/CSS/JS
```

SQLite is the single source of truth.

- JSON is a generated search/rendering artifact.
- HTML is a generated static-site artifact.
- Generated artifacts should not be treated as manually authoritative data.
- Prefer data correctness over aggressive automatic inference.
- Incorrect mappings should be fixed through explicit merge operations.

## Expected CLI Surface

The specification describes these commands:

```bash
ntindex crawl
ntindex build
ntindex update
ntindex merge game <old_id> <new_id>
ntindex merge character <old_id> <new_id>
```

Command intent:

- `crawl`: collect NeonTeam videos, parse titles, create missing games and characters, and record new videos.
- `build`: read SQLite and generate static JSON and HTML output.
- `update`: run crawl, then build.
- `merge game`: consolidate duplicate game records.
- `merge character`: consolidate duplicate character records within the same game.

## Data Model Principles

Expected tables:

- `games`
- `characters`
- `videos`

Important constraints:

- Game names are unique.
- Character names are unique only within a game.
- The same character name may exist in different games.
- Video links are unique.
- Character merge is valid only when old and new characters belong to the same game.

The expected lookup index is:

```sql
CREATE INDEX idx_videos_swap
ON videos (
    game_id,
    source_id,
    target_id
);
```

## Search And Build Behavior

- Game pages are generated per game.
- Character pages are not generated by default.
- Character filtering and `A as B` lookup should be handled client-side using `search.json`.
- Static hosting without an application server is a project goal.

## Validation Expectations

For changes that affect behavior:

- Prefer a small test or reproducible check before changing code.
- Verify parsing against the title format in `SPEC.md`.
- Verify database writes preserve uniqueness and foreign-key expectations.
- Verify generated JSON/HTML can be rebuilt from SQLite.
- If full validation is not possible, report exactly what was and was not checked.

Do not claim behavior is verified unless it was actually run or inspected.

## Documentation Expectations

When adding or changing modules, keep documentation concise and useful.

Document only what helps future agents or maintainers understand:

- File purpose.
- Inputs and outputs.
- Important dependencies.
- Non-obvious assumptions.

Avoid broad architecture rewrites unless the user asks for them.
